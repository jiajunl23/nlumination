# v3 — NLumination

> v2 ended with a 3-agent pipeline (A1 emotion ‖ A2 numeric image stats → A3 action) that handled "vintage faded film" and "moody and contemplative" through structured Groq calls under a 100/day shared budget. v3 turns that pipeline into something that **remembers, sees, and reaches into a real library of film looks** — and ships the security perimeter the v2 prototype was missing.

The constraint going in: keep v2's free-tier cost envelope (≤100 LLM calls/day per user, $0 marginal cost) while adding (a) refinement-aware dialogue, (b) actual vision input to A2, and (c) a way for power users to break past the daily cap with their own Groq key. All three landed in `ab7a6f9` as Phases A/B/C, then five more shipping rounds layered LUT retrieval, visual-grounding, and a security audit on top.

---

## What shipped

### 1. Conversation context (Phase A)

Every editor session now keeps a parallel `turnHistory` array of structured `TurnRecord`s — one per successful submit:

```ts
type TurnRecord = {
  prompt: string;
  paramsBefore: GradingParams;
  delta: LLMDeltaT;
  paramsAfter: GradingParams;
  timestamp: number;
};
```

The client persists it under `localStorage["nlumination.turnHistory:v1"]` and POSTs the most recent N turns to `/api/nlp/interpret` on every call. Server-side `trimHistory()` re-truncates to a 6000-char budget before any LLM call so an unbounded session can't blow the input window. `summariseHistory()` renders each turn into ~160 chars (`"T: <prompt> | Δ: <changed-fields> | params changed: <diff>"`), which means ~37 turns survive the cap — enough for a full session, hard ceiling for safety.

A1's emotion prompt and A3's action prompt both append the rendered trail. The result: prompts like *"actually a touch less dramatic"* now produce **negative deltas** that walk back the previous turn, because A3 sees both the post-history state AND the chain of intent that got there. Verified in Playwright: the prompt above, two turns into a teal-orange session, produced `-2 contrast, -4 shadows, -2 clarity` — exactly the reverse direction of the prior `+10/+10/+4`.

### 2. BYO Groq key (Phase B)

Power users paste their own `gsk_*` key into a settings popover. Client validates format (`/^gsk_[A-Za-z0-9]{20,}$/`), stores in `localStorage["nlumination.groqApiKey"]`, and ships it on every NL request via an `X-Groq-Key` HTTP header. Server:

- Reads the header **before** the env-key check, so the app works even when `GROQ_API_KEY` isn't configured server-side.
- Re-validates the format; malformed → `400 key_format`.
- Builds a per-request `OpenAI` client via `getGroqForKey(apiKey)` — never cached, never stored.
- **Skips both `getRemaining()` and `incrementUsage()`** when a BYO key is present. Their key, their tokens.
- Returns `quota: { unlimited: true }` instead of `{used, limit}`, so the chat panel swaps the badge to `🔑 Using your key`.

Security stance, documented in `app/api/nlp/test-key/route.ts`:
- Header-only (never accept in body) keeps the key out of access logs that capture POST bodies.
- `console.error` lines log `err.status + err.message` only — `grep gsk_ /tmp/server.log` is empty.
- Format check before the Groq call so a typo doesn't even hit Groq.
- `/api/nlp/test-key` is a separate endpoint with a tighter `maxDuration: 15` and tinier surface area — it does a 1-token completion just to confirm the key authenticates, then returns 200/401/502.

### 3. VLM image analyst (Phase C)

A2 in v2 read 7 numbers (mean luminance, p05/p95, channel means). It could tell "dim" from "bright" but not "portrait at golden hour" from "neon-lit street". v3 routes A2 through **Groq Llama-4-Scout** (`meta-llama/llama-4-scout-17b-16e-instruct`) — a 17B-active MoE with native vision — when an image URL is available, with the original numeric path as a hard fallback.

Image data path:
- **Saved photo** (has Cloudinary publicId): pass `originalUrl(publicId)` directly. Groq's egress fetches it from the CDN. Zero client→server bloat.
- **Fresh upload** (not yet saved): client downsamples ImageBitmap to 384×384 via OffscreenCanvas, exports JPEG @ 0.8, base64-encodes, ships as `data:image/jpeg;base64,...`. ~30 KB request bloat — acceptable.

VLM prompt is tightly scoped: ≤40 words, one sentence covering subject type / lighting / color-mood character / one obvious flaw. No identity, no OCR. Numeric stats ride along as a cross-check anchor since VLMs occasionally misjudge cast on heavily-saturated photos.

Cost: Llama-4-Scout sits in an **independent 500K TPD bucket** on Groq's free tier — does not compete with gpt-oss-20b's 200K bucket. Per VLM call ≈ 810 tokens (600-800 image + 80 system + 30 user + 50 output + 50 reasoning) → ~617 VLM calls/day before paid territory, vs the 100 NL requests/day app cap. Cost-neutral on free tier.

Fallback: any 4xx from the VLM (model retired, image rejected, payload too big) silently falls through to the numeric stats path. The agents pipeline never loses A2 entirely.

### 4. Retrieval-augmented LUT tool selection (the meat of v3)

The single biggest behavior change in v3. A3 can now seed a real 3D color LUT alongside its slider deltas, picked from a 137-entry license-clean library by cosine retrieval over the user prompt.

**The library** (`public/luts/`, 137 PNGs, ~4 MB total):
- **51 × MIT** — t3mujinpack film simulations (Kodak Portra/Ektar/Gold, Fuji Velvia/Provia/Astia/Superia, Ilford Tri-X/HP5/Delta, Agfa APX, Polaroid B&W, etc.)
- **86 × CC-BY-SA-4.0** — RawTherapee Film Simulation Collection cherry-picks: 33 CreativePack-1 cinematic looks (teal-orange / bleach-bypass / foggy-night / horror-blue), 18 Polaroid color variants with Cold/Warm/index combinations, 2 Lomography (Redscale, X-Pro Slide), plus various push/pull and cross-process variants.
- License attributions and ShareAlike scoping documented in `public/luts/LUTS_LICENCE.md`.

**HaldCLUT + WebGL TEXTURE_3D pipeline**:
- Every LUT ships as a 216×216 PNG in Hald-6 layout (`36³ effective LUT`, `x = R + (G mod 6)·36, y = G/6 + B·6`).
- Client-side decoder (`lib/webgl/hald-clut.ts`) fetches the PNG, draws into an OffscreenCanvas, unrolls into a Float32Array 3D cube, with an in-flight dedup cache so repeated `setLut` calls don't redecode.
- `lib/webgl/textures.ts` uploads the cube to GPU as `gl.TEXTURE_3D` with LINEAR filter + CLAMP_TO_EDGE on S/T/R — hardware trilinear interpolation, ~5 ms for an 8 MP image vs ~50 ms for a CPU equivalent.
- `lib/webgl/shaders/lut.frag.ts`: `outColor = vec4(mix(src, looked, u_opacity), 1.0)` — the LUT applies as a separate pass *before* the slider stage, so slider deltas A3 emits alongside `lutId` are fine-tuning on top of the graded image.

**Build pipeline**:
- `scripts/build-luts.ts` walks `/tmp` source bundles and emits the manifest skeleton with category / film-stock / tag fields per entry.
- `scripts/build-luts-haldaware.py` resamples vendor Hald-12 (1728×1728, 144³ effective) → Hald-6 (216×216, 36³). Critically, this uses **cube-space trilinear** resampling — `magick -resize -filter Lanczos` produces visually plausible but mathematically corrupt output because spatial blur crosses HaldCLUT cell boundaries that represent unrelated LUT inputs. Replaced after a "the photo looks weird" bug report: `tealorange` at input (0,0,0) was returning (13, 86, 21) instead of (0, 2, 21) — i.e. injecting greens into every black pixel. After the cube-aware rebuild, sampled values match vendor reference exactly.

**Retrieval** (`lib/nlp/lut-retrieve.ts`, `lib/nlp/embedding.ts`):
- Indexed with **Voyage AI `voyage-3-lite`** (512-d, asymmetric pooling). Voyage was chosen over HuggingFace's free Inference API after the latter started returning 503s mid-eval; Voyage's free tier is 50M tokens/month without a credit card, which we'll never hit at ~30 tokens per query.
- Asymmetric retrieval: corpus embedded with `input_type="document"`, queries with `input_type="query"`. Mixing the pooling sides silently degrades cosine quality by 10-20%.
- Manifest cached at module init into a flat `Float32Array(137 × 512)` + precomputed `Float32Array(137)` of L2 norms. Cosine reduces to a dot product / qNorm × dNorm, brute-force across all 137 entries. No ANN index — at 137 entries an HNSW tree's setup overhead exceeds the saving.
- `retrieveLutsStrict` (strict variant) throws on Voyage / manifest errors so the trace records `ok: false`. Graceful variant returns `[]`. Both exposed; the graph uses Strict.

**A3 prompt extension** (`lib/nlp/agent/prompts.ts:181-220`):
The action prompt gained a 40-line "LUT SEEDS" section that teaches A3 when to set `lutId` (score ≥ 0.35 AND prompt is a style intent), when not (numeric tweak, all scores < 0.30, "don't change colors" override), and how to scale `lutOpacity` (0.6-1.0 strong, 0.3-0.5 hint). User prompt includes a JSON array of top-3 candidates: `[{"id":"rt-color-creativepack-1-tealorange","desc":"classic Hollywood teal-orange — water and shadows pulled to deep teal-cyan...","score":0.78}, ...]`. Inventing an id outside this list = silent drop.

**Schema and pipeline plumbing**:
- `LLMDelta` (`lib/nlp/llm-schema.ts`) gained `lutId: string | null` and `lutOpacity: 0..1`.
- `mergeDelta` treats `lutId: ""` as `null` (clear-the-slot semantics).
- `graph.ts` runs 4 nodes in parallel: A1 ‖ A2 ‖ `lutRetriever` (1 Voyage embedding call, ~200 ms warm) → A3. Total wallclock = max of the three independent nodes ≈ 600-800 ms, not the sum.
- Failure modes in `lutRetriever`: Voyage 429 → trace records the error, A3 runs without LUT seeds, request still succeeds.

**Grade-mode toggle** (`components/editor/ChatPanel.tsx`, persisted under `nlumination.gradeMode:v1`):
- `"auto"` — A3 decides per the rules above.
- `"lut"` — A3 must seed with the top candidate even if score is mediocre.
- `"slider"` — A3 must NOT emit `lutId`; legacy v2 25-slider behavior.

This toggle is the agents-mode escape hatch: users who don't want LUT influence at all can pin Slider mode; users who specifically want a film look can pin LUT mode to skip the heuristic.

### 5. Visually-grounded LUT descriptions

The initial round of `scripts/enrich-lut-descriptions.py` wrote descriptions from a hand-curated film-stock knowledge base ("Kodak Portra 400 — the modern wedding-and-editorial classic, warm golden skin, milky highlights"). Solid for known stocks; weak for variants where the only differentiator was a number — `bleachbypass1` / `bleachbypass2` / `bleachbypass3` all read as "Bleach-bypass variant N" with no observable difference in retrieval embedding space.

The fix (commit `d8ee874`):

1. **Tour script** — added a dev-only `window.__setLut(id, opacity)` debug hook in `EditorRoot`, gated by `NODE_ENV !== 'production'`.
2. **Browser-driven LUT walk** — Playwright MCP iterates all 137 entries, applying each via `__setLut`, capturing the WebGL canvas to JPEG via `canvas.toDataURL`, and POSTing each to a tiny throwaway local writer server (`scripts/lut-tour-writer.mjs`) on `127.0.0.1:5555`. Total time: ~5 min, ~15 MB of screenshots under `lut-tour/`.
3. **HTML index** — `scripts/build-lut-tour-html.mjs` generates a grid view (`lut-tour/index.html`) with each card showing id / bundle / category / description / tags next to the rendered thumbnail.
4. **Visual inspection** — I read each screenshot, comparing against the baseline, and rewrote 86 descriptions to capture what each LUT *actually does on a photo* rather than its canonical film-stock recipe. `bleachbypass1` is now "extreme bleach-bypass — sky blown to near-white, deep cool cyan-blue cast through midtones, near-monochrome low-saturation, harsh hi-contrast"; `bleachbypass4` is "strong bleach-bypass — midtones pushed toward monochrome silver-grey while sky retains gentle warmth; cinematic Saving-Private-Ryan look". Direction variants got the same treatment: every Cold variant has "cool cyan-teal cast" verbiage, every Warm has "warm cream-amber wash", every XPRO has "extreme blue-cyan / magenta highlights / surreal lo-fi lomography vibe".
5. **Re-embed** via `scripts/embed-luts.ts` (one 65 s pre-clear pause + 22 s between batches to respect Voyage's 3 RPM free-tier limit on small accounts).
6. **Smoke-retrieve verification** — `scripts/smoke-retrieve-visual.ts` runs 12 direction-named prompts. Result:
   - `"cold cyan polaroid cinematic instant"` → top 3 = all 3 Cold polaroid variants
   - `"warm cream amber faded polaroid"` → top 3 = all 3 Warm polaroid variants
   - `"cross processed surreal lo-fi blue magenta"` → top 3 = all 3 XPRO LUTs
   - `"near-monochrome cool grey-blue dystopian sci-fi"` → futuristicbleak4 @ 0.792 (verbatim description match)

### 6. On-canvas affordances — Start over + LUT pill

Two small UI fixes shipped late in v3 after Playwright sessions surfaced friction:

**Start over** (`2496fcf`) — the only way to abandon an edit and pick a different photo used to be navigating back to home then re-entering `/editor`. New pill in the canvas overlay top-left (next to filename) clears the photo, params, LUT, chat history (memory + localStorage), and `?photoId=` from the URL in one click. Preferences (Agents/LLM mode, grade mode, BYO key) are preserved — they're not photo-scoped. ChatPanel + Canvas are remounted via a `sessionKey` to wipe internal refs.

**LUT-applied indicator pill** (`e871507`) — when `params.lutId` is set, an accent-colored pill appears top-right of the canvas: `🎨 LUT · <humanized-name> [opacity%]` with an inline `×` to remove the LUT while keeping slider deltas. `humanLutName()` strips bundle/category prefixes (`rt-color-`, `t3-color-negative-`, `creativepack-1-`) and dedupes consecutive tokens so `rt-color-polaroid-polaroid-669-cold-3` reads as "polaroid 669 cold 3". Before this pill, the only signal a LUT was active was the trace line `🎨 Retrieved 3 LUTs (top: ...)` — which describes retrieval, not selection.

**BYO key trigger redesign** (`4ce9952`) — the popover trigger used to be a generic `Settings` gear icon, indistinguishable from a "preferences" button. Now an h-7 pill with the `KeyRound` icon + a text label that switches between `BYO key` (idle, muted border) and `Your key` (accent border + bg + text, when a key is saved). Tooltip text is explicit per state: *"Bring your own Groq key to bypass the shared 100/day quota"* idle, *"Using your Groq key — unlimited daily quota. Click to manage."* active. Same shape and rhythm as the surrounding Auto/LLM/Agents mode pills, so the chat-panel header now reads as a row of equally-weighted controls.

**Second sample image** (`6d1413e`) — the existing `/sample.jpg` is a warm-lit night airport scene; good for testing warm-cast handling but doesn't exercise wide tonal range or mixed cool/warm content. Added `/sample-2.jpg` (Dali Erhai lakeside at dusk, 1920×1080, 432 KB) — sky/water gradient covering most of the luminance range, cool blue water with warm sunset horizon (real teal-orange material), bright pastel sky tones to test highlight retention. This is the same image used to capture the `lut-tour/` comparison screenshots, so users can replicate that gallery's grading exactly. DropZone refactored to take a `SampleSpec[]` array with `loadingSampleId` state so only the clicked pill shows "Loading…" while the other stays interactive. UI is now "Or try a sample: [Airport] [Lakeside]" instead of one anonymous button.

### 7. Security perimeter

A late-v3 security audit covered 7 layers (secrets, auth surfaces, AI inputs, env scoping, CI gates, monitoring, supply chain). Findings turned into 4 commits:

- **`83dee55`** — `imageUrl` SSRF defence: validator in `lib/nlp/image-url.ts` accepts only `https://res.cloudinary.com/<our cloud>/...` or `data:image/(jpeg|png|webp);base64,*`. Before this, a logged-in user could weaponize the VLM endpoint as a free open proxy for any URL or probe internal/cloud-metadata IPs through Groq's egress. Same commit also locked Cloudinary signed uploads to `resource_type=image` + `allowed_formats=jpg,jpeg,png,webp` (SVG-with-embedded-JS was previously not rejected), added the X-Frame-Options/nosniff/Referrer-Policy/Permissions-Policy headers via `next.config.ts`, and tightened the catch-all error log in `/api/nlp/interpret` to log only `err.name + err.message.slice(0, 200)`.
- **`6005449`** — `next 16.2.4 → 16.2.6`. `pnpm audit`: 15 vulnerabilities (7 high incl. Middleware/Proxy bypass in App Router) → 2 moderate, both build-time-only.
- **`87221f7`** — `AGENTS.md` deny list (agents cannot read `.env*`, modify `proxy.ts` matcher, remove `auth.protect()` calls, push to non-origin remotes, skip hooks). CI workflow gained `pnpm audit --audit-level=high`, `gitleaks/gitleaks-action@v2` for secret scanning on every push, and `actions/dependency-review-action@v4` on PRs to block new high-severity transitives.

Auth posture verified by audit: every API route imports `requireDbUser()` and calls it inside the handler; every DB query scopes by `userId`; Clerk middleware (`proxy.ts`, Next 16's renamed `middleware.ts`) protects all `/api/*` routes via `auth.protect()`. Rate limiting beyond the daily 100/user quota is deferred to v4 — abuse cost is capped at `100 × N accounts`, and account-creation throttling is set in the Clerk dashboard rather than code.

## Three Groq landmines from v2 still relevant, one new one

v2 wrote down three pitfalls (`tools` ⊥ `response_format: json_schema`, `strict:false` silent strictness, `gpt-oss-20b` reasoning tokens count toward billing). All still apply. v3 added one:

**4. Llama-4-Scout is a preview model — its id can change without notice.** The Groq docs flagged this as "preview" when we wired it; `imageMoodAnalyst` catches `OpenAI.APIError` and falls through to the numeric stats path so a model retirement doesn't break A2 entirely. Verified by manually pointing `GROQ_VLM_MODEL` at a bogus id during testing — the 400 propagated through, the trace recorded `vlm error: 400`, the numeric path kicked in, A3 still produced a delta.

## Numbers

Free-tier budget shape:
- gpt-oss-20b (A1, A3, single-shot fallback): 200K TPD shared across users.
- llama-4-scout (A2 VLM): independent 500K TPD bucket. Cost-neutral, doesn't compete.
- voyage-3-lite (LUT retrieval): 50M tokens/month free, ~30 tokens/query → 1.6M queries/month headroom.
- App-side per-user cap: 100/day, enforced via `llm_usage` table atomic upsert.

Per-request token cost (agents mode):
- A1 (emotion): ~280 tokens
- A2 VLM (image): ~810 tokens
- Retrieval (Voyage embedding): ~30 tokens (separate bucket)
- A3 (action, with LUT candidates injected): ~1800 tokens (was ~1769 in v2 without LUT seeds)
- **Per-request total** ~2890 tokens against the 200K TPD bucket → ~69 agents-mode requests/day on the shared key before billing.

Per-request token cost (llm mode):
- Single-shot (no analyst, no retrieval): ~543 tokens — unchanged from v2 post-trim.

Retrieval latency (Voyage warm): 150-250 ms. Cosine over 137 × 512 floats: < 1 ms. Manifest load on cold start: ~50 ms (1.5 MB JSON parse). HaldCLUT decode + GPU upload: ~80 ms first time per LUT, ~5 ms thereafter (cached).

LUT library: 137 entries × 216×216 PNG ≈ 4.0 MB on disk. Manifest with embeddings: 1.55 MB JSON.

## What v3 proves

v2 made the case that a multi-agent pipeline could interpret one user prompt against one image-context snapshot. v3 makes the case that **the same pipeline can carry intent across turns, reach into a real-world asset library through dense retrieval, and accept actual vision input — without breaking the cost envelope or the security perimeter**.

The retrieval piece is the load-bearing one. Slider deltas can encode "warmer" and "more contrast" cleanly. They can't encode "Wong Kar-wai red-green nostalgia" or "Polaroid Time Zero expired warm fade" in a few +/-N adjustments. Either the model fakes it (8-14 sliders flapping at different magnitudes, the eval graders flag it as over-correction) or it picks the right LUT and the sliders fine-tune the result. v3 gives the model that option, with the prompt-engineered guidance to know when to use it.

The visual-grounding rewrite of LUT descriptions is the unsung half of this. Before it, "cold polaroid" and "warm polaroid" retrieved the same Polaroid base entries regardless — the canonical film-stock copy described what Polaroid 669 IS, not what Cold vs Warm DOES. After 86 manual rewrites with screenshots in hand, direction-named prompts retrieve their direction-specific variants cleanly. RAG is only as good as its corpus copy.

## Where it stands at end of v3

- **Agents mode (BYO key OR shared 100/day)** — A1 emotion + A2 VLM + LUT retrieval (top-3) → A3 with LUT seed + 30-field grading delta.
- **LLM mode** — single-shot fallback path, ~543 tokens/call. Still works, still cheap.
- **Slider mode** — legacy v2 25-slider behavior, fully preserved.
- **GradeMode**: auto (A3 decides) / lut (force pick) / slider (no LUT). User-toggleable.
- **Conversation memory** — full session history, server-trimmed to 6000 chars, persisted across reloads.
- **Vision-aware** — Llama-4-Scout reads the photo when available, falls through to numeric stats otherwise.
- **137 LUTs** — license-clean, visually-grounded descriptions, Voyage-embedded, served as Hald-6 PNG decoded into WebGL TEXTURE_3D.
- **Security**: 0 high CVEs, SSRF/upload validation, security headers, agent deny list, CI gates.

Next likely directions (v4):
- Critic-loop agent (A4 VLM judges the rendered result vs prompt intent; A3' revises once if needed).
- Token-based quota instead of call-based.
- 5-10 hand-crafted signature-look LUTs (Wong Kar-wai, Wes Anderson, Roger Deakins) since the film-stock library is saturated.
- Sentry / structured LLM-call logging for production observability.
- Rate-limiting beyond daily quota (per-IP+per-minute via Upstash).
